01. Data Architecture phase 1 ▶

Визуальная схема стека

Интерактивная схема по слоям (как шпаргалка data-stack). Каждый слой раскрывается.

Контент по каждому слою

Sources: PostgreSQL (продакшн), Kafka (ставки, события real-time), GA4+GTM (web), Adjust (mobile), CRM, external APIs (odds feeds, payment providers). По каждому: 2-3 предложения зачем, что из него приходит в DWH.

Ingestion: Airbyte (основной, self-hosted) + Debezium/CDC (real-time из продакшна через Kafka Connect). Альтернатива: Fivetran (managed, дороже). Пояснение: почему self-hosted, когда оправдан managed.

Storage: ClickHouse (основной — быстрая аналитика, популярен в СНГ/iGaming) + GCS data lake (холодные/сырые данные, Parquet). Альтернативы: BigQuery (serverless, дороже на объёме), Snowflake (западный стандарт). Trade-off: ClickHouse быстрее и дешевле, но нужна DE-команда для поддержки.

Transformation: dbt Core. Три слоя с примерами: staging (1:1 с источником, очистка) → intermediate (бизнес-логика, joins) → marts (витрины для BI/DS). Пример одной цепочки: stg_bets → int_bets_enriched → mart_daily_kpi. Альтернатива: Spark (для тяжёлых batch-задач).

Orchestration: Apache Airflow. Пример DAG: extract → load → dbt run → dbt test → alert. Объяснение: почему не cron (зависимости, retry, UI). Альтернативы: Dagster, Prefect.

DQ & Governance: dbt tests (первая линия) + Great Expectations (сложные проверки) + DataHub (catalog). Подробнее — в секции 06.

BI: Metabase (основной, self-service) + Looker Studio (маркетинг, бесплатный). Альтернативы: Tableau (мощный, дорогой), Power BI (MS-стек), Looker (semantic layer). Подробнее — в секции 12.

ML: Python/Jupyter (разработка) + MLflow (lifecycle) + Feast (feature store). Подробнее — в секции 09.

Reverse ETL: Census (DWH → CRM, рекламные платформы). Use case: churn score из модели → сегмент в CRM → персональный бонус.

Формат карточки инструмента

Название → Что делает (1-2 предложения) → Почему выбрали (1 предложение) → Альтернатива и почему не она (1 предложение) → Ключевой trade-off

02. Org Structure phase 1 ▶

Орг-схема

Визуальная диаграмма: Director → 4 лида → команды. Кликабельная — каждый узел раскрывает детали.

4 стрима — детализация

Data Engineering (5 чел):
Lead DE + 2 Data Engineers (пайплайны, DWH) + 1 Analytics Engineer (dbt-модели, DQ) + 1 Platform/DevOps (Airflow, ClickHouse, мониторинг)
Зона ответственности: ingestion, DWH, инфраструктура, data quality на уровне пайплайнов, on-call
Взаимодействует с: Backend (доступ к данным), DevOps (инфра), Analytics (требования к витринам)
Ключевые deliverables: пайплайны работают 24/7, данные свежие к 7:00, SLA uptime 99.5%

Analytics (8 чел):
Lead Analyst + 3 Product Analysts (sports, casino, payments) + 2 Marketing Analysts (acquisition, retention/CRM) + 1 Financial Analyst (P&L, unit economics) + 1 BI Developer (дашборды, semantic layer)
Зона: отчётность, исследования, ad-hoc запросы, когортный анализ, воронки
Взаимодействует с: Product, Marketing, Finance, C-level
Ключевые deliverables: дашборды, еженедельные/ежемесячные отчёты, ad-hoc за 3 дня, исследования за 2 недели

Data Science (4 чел):
Lead DS + 1 ML Engineer (деплой моделей, MLOps) + 1 DS (LTV prediction, churn, рекомендации) + 1 DS (fraud detection, risk scoring, anomaly detection)
Зона: predictive models, A/B тестирование (методология + анализ), experimentation platform
Взаимодействует с: Product (use cases), DE (данные, feature store), Risk/Compliance (fraud)
Ключевые deliverables: модели в проде, A/B отчёты, fraud scoring

BI Engineering (3 чел):
Lead BI + 1 BI Developer (Metabase, сложные дашборды) + 1 Data Literacy Specialist (обучение бизнеса, self-service, документация)
Зона: semantic layer, стандартизация отчётности, self-service инструменты, data literacy программа
Взаимодействует с: все бизнес-функции (end users дашбордов)
Ключевые deliverables: self-service покрытие, обучение, стандартизированные шаблоны

Дополнительно

RACI-матрица: кто Responsible / Accountable / Consulted / Informed по ключевым процессам (новый дашборд, новый пайплайн, новая модель, инцидент)

Skill matrix: таблица навыков по ролям — SQL, Python, dbt, BI, statistics, ML, communication, domain knowledge. Уровни: basic / proficient / expert.

Career ladder: Junior → Middle → Senior → Lead → Head/Director. Что ожидается на каждом уровне.

dbt layers

Staging (stg_): 1:1 с источником. Очистка типов, переименование, дедупликация. Пример: stg_betting__bets, stg_payments__deposits

Marts (mart_): финальные витрины для BI и DS. Пример: mart_daily_kpi, mart_player_lifetime, mart_cohort_analysis, mart_marketing_performance

Для каждого слоя: 1 конкретный пример SQL-модели (упрощённый, 10-15 строк)

04. Event Taxonomy phase 1 ▶

Naming convention

Формат: {domain}_{object}_{action}

Domains: player, session, marketing, system
Objects: bet, deposit, withdrawal, bonus, session, page, registration, kyc
Actions: placed, completed, started, viewed, settled, issued, expired

Примеры: player_bet_placed, player_deposit_completed, session_page_viewed, player_registration_completed, player_kyc_verified, player_bonus_issued

Event schema (стандартная структура)

Каждое событие содержит:
Обязательные: event_name, event_id (UUID), player_id, timestamp (ISO 8601), session_id, platform (web/ios/android)
Контекстные (auto): geo_ip, device_type, app_version, page_url
Специфичные: зависят от события (amount, game_id, method и т.д.)

Пример JSON для player_bet_placed и player_deposit_completed — полная структура

Tracking plan

Таблица: event_name | trigger (когда срабатывает) | parameters (список) | source (frontend/backend/server) | owner (команда) | status (live/planned/deprecated)

10-15 ключевых событий с заполненными строками

Процесс изменений

Как добавить новое событие: request → review (analytics + DE) → spec → implementation → QA → deploy. Кто аппрувит, как версионируется.

06. Data Governance phase 2 ▶

Контент

Роли и ответственности:
Data Owners по доменам: Head of Product (betting data), CFO (finance), CMO (marketing), Head of Compliance (player PII). Для каждого: что именно «владеет», какие решения принимает.
Data Stewards: назначенный аналитик/DE на каждый домен. Пример: steward витрины mart_daily_kpi — кто это, что проверяет.
Governance Council: состав, частота встреч (quarterly), типичная повестка, пример решения.

Data Catalog (DataHub): пример карточки таблицы — название, описание, owner, steward, freshness, lineage, теги, популярность (кто запрашивает).

Data Dictionary: стандартизированные определения ключевых терминов. Active Player = сделал ставку за последние 30 дней. FTD = first-time depositor. Процесс: как предложить изменение определения, кто аппрувит.

Access policies (RBAC): матрица ролей: Analyst / DS / BI Dev / Lead / Director × домены данных (betting, finance, player PII, marketing). Row-level security пример: маркетинг видит когорты без PII.

DQ framework: 5 измерений с примером теста на каждый. Completeness: «% NULL в поле payment_method < 1%». Accuracy: «actual RTP vs theoretical ±2pp». Timeliness: «mart обновлён до 7:00 UTC». Consistency: «sum(fct_bets.amount) = mart_daily_kpi.turnover ±0.01%». Uniqueness: «0 дублей по bet_id».

Data lifecycle: retention policy. Горячие (3 мес, ClickHouse) → тёплые (1 год, ClickHouse cold storage) → холодные (5 лет, GCS Parquet). Регуляторные требования: данные ставок 5+ лет.

09. MLOps phase 2 ▶

Контент

ML lifecycle: визуальная схема — Problem framing → Data prep → Training → Validation → Deploy → Monitor → Retrain

11. Incident Response phase 2 ▶

Контент

Severity levels: P1 (критичная витрина/pipeline down, CEO report broken, fraud scoring offline) → P2 (некритичный дашборд stale, DQ test failed на некритичном поле) → P3 (minor DQ issue, documentation outdated)

Response protocol: визуальная схема: Detect (alert) → Acknowledge (assign owner, ETA) → Diagnose (root cause) → Fix → Communicate (stakeholders) → Post-mortem

Post-mortem template: заполненный пример. Title, Date, Severity, Duration, Impact (кто затронут, какие решения не могли принять), Timeline (minute-by-minute), Root cause, What went well, What went wrong, Action items (с owner и deadline)

Пример инцидента: «Понедельник 7:30. mart_daily_kpi не обновился. CEO пишет: цифры за выходные выглядят неправильно. Root cause: Airflow worker out of memory из-за spike в данных после финала Лиги Чемпионов. Fix: увеличили memory limit, добавили партиционирование. Preventive: алерт на OOM, автоскейлинг.»

13. A/B Testing Framework phase 3 ▶

Контент

HADI-цикл: визуальная схема Hypothesis → Action → Data → Insight → (loop). Пояснение каждого шага.

Бэклог гипотез: таблица-шаблон: ID, гипотеза (формат: «если [изменение], то [метрика] [направление] на [MDE], потому что [обоснование]»), primary metric, MDE, ICE score, status (backlog/running/done), owner. 5-7 заполненных примеров для iGaming.

Чеклист запуска теста: 8 пунктов — гипотеза, метрики (primary/secondary/guardrail), sample size расчёт, рандомизация, длительность, стейкхолдер sign-off, no-peek commitment, rollback plan.

Шаблон отчёта: гипотеза → setup (sample size, duration, split) → результат (CI, p-value, effect size) → сегментация → рекомендация (roll out / iterate / kill) → learnings.

Полный кейс: тест упрощения формы депозита. Гипотеза → расчёт → запуск → результат → решение. С конкретными цифрами (фейковыми).

14. Management Practices phase 3 ▶

Контент

Sprint structure: 2-недельные спринты. Визуальная timeline: Planning (пн, 1ч) → Daily standups (15 мин) → Mid-sprint check → Review (пт 2й недели, 1ч) → Retro (30 мин). 70% roadmap / 30% ad-hoc buffer.

Meeting cadence: визуальная сетка-календарь. Director ↔ Leads: weekly 1-on-1 (30 мин). Director ↔ C-level: bi-weekly (KPI, roadmap, blockers). All-hands data: monthly (результаты, планы, обучение). Governance Council: quarterly. Cross-functional (Product/Marketing/Finance): weekly.

1-on-1 с лидом: структура: People (кто горит/выгорает, найм) → Process (что буксует, bottlenecks) → Strategy (OKR progress, корректировки). 30 мин, лид инициирует повестку.

OKR framework: пример квартальных OKR для data-команды. Objective: «Обеспечить data-driven принятие решений». KR1: 100% критичных витрин обновляются с SLA. KR2: self-service покрытие 80% ad-hoc запросов. KR3: LTV-модель в проде с MAPE <20%.

Backlog management: как приходят запросы (Jira/Linear form), как приоритизируются (RICE), как коммуницируется статус (Slack bot + weekly digest).

Knowledge management: структура wiki — Architecture, Metrics, Processes, Runbooks, Onboarding. Правило: «если объяснил в Slack — запиши в wiki».
