Onboarding

Путь нового сотрудника data-команды: от получения доступов до самостоятельных задач. Buddy, data tour, первый PR — за 30 дней.

Этот гайд покрывает общий onboarding для всех ролей в data-команде. Специфика по стримам (DE, Analytics, DS, BI) — через buddy и лида.

День 1–3: Доступы и знакомство

Первые дни — административный setup и знакомство с командой. Цель: к концу дня 3 у тебя есть все доступы и понимание, кто чем занимается.

▶ Buddy (Day 1)

Каждому новичку назначается buddy — опытный сотрудник из того же стрима (или смежного). Buddy — не руководитель, а навигатор.

Что делает buddyЧего buddy НЕ делает
Отвечает на «глупые» вопросы без осужденияНе оценивает performance
Помогает разобраться в контексте задачНе назначает задачи
Code review первых PRНе заменяет лида
Обед/кофе в первую неделюНе отвечает за retention

Buddy назначается лидом стрима до выхода нового сотрудника. Длительность: 1 месяц, далее по желанию.

▶ Знакомство с командой (Day 1–2)
  • Day 1, 11:00 — Welcome-встреча с лидом стрима (30 мин). Обзор команды, приоритетов, ожиданий.
  • Day 1, 14:00 — Знакомство с buddy. Совместный обед или кофе.
  • Day 2, 10:00 — Встреча с Director of Data (15 мин). Миссия data-функции, big picture.
  • Day 2, 11:00 — Круг знакомств: 15-минутные 1-on-1 с каждым лидом стрима.
  • Day 2–3 — Присутствие на ближайшем daily standup и sprint review как наблюдатель.
▶ Обязательное чтение (Day 2–3)
  • Metrics Dictionary — ключевые метрики бизнеса (NGR, GGR, LTV, Churn)
  • Event Taxonomy — naming convention, как устроены события
  • Data Governance — ownership, доступы, DQ framework
  • Wiki: Data Dictionary — стандартные определения терминов
  • Wiki: Team Norms — code style, PR process, communication rules
Не пытайся запомнить всё за 2 дня. Цель чтения — сформировать mental map: «знаю, что это существует и где найти». Детали придут с практикой.

Неделя 1: Data Tour

Главное событие первой недели — «Data Tour»: ты проходишь путь данных от события до дашборда. Это даёт целостное понимание архитектуры.

▶ Data Tour: от события до дашборда

Вместе с buddy проходишь end-to-end путь конкретной метрики (обычно NGR):

1. Источник
Betting Engine генерирует событие player_bet_placed. Смотрим формат в Event Taxonomy.
2. Ingestion
Событие попадает в Kafka → Airbyte загружает в ClickHouse raw layer. Смотрим Airflow UI — DAG ingest_kafka_bets.
3. Transformation
dbt модели: stg_betting__bets → int_bets_enriched → mart_daily_kpi. Открываем код в GitHub, запускаем dbt run --select mart_daily_kpi на staging.
4. Quality
dbt tests на mart_daily_kpi: uniqueness, not_null, referential integrity. Смотрим результаты в Airflow.
5. Dashboard
Открываем Metabase → CEO Daily Pulse → видим NGR, который мы только что проследили от события до цифры. Подробнее в Dashboards.

Полная схема lineage: Data Lineage.

▶ Архитектурный обзор

С buddy или лидом DE проходишь Data Architecture:

  • 9 слоёв стека: что каждый делает, какой инструмент, почему выбран
  • Как слои связаны между собой
  • Где логи, мониторинг, алерты

После обзора — самостоятельно пройдись по Data Model: ERD, таблицы, поля.

▶ Первый SELECT

К концу недели 1 ты должен самостоятельно написать запрос к DWH. Задание от buddy:

-- Задание: посчитай основные метрики за последние 7 дней
SELECT
    dt,
    active_players,
    turnover,
    ggr,
    ngr
FROM mart_daily_kpi
WHERE dt >= today() - 7
ORDER BY dt DESC;

Цель: убедиться, что доступы работают и ты понимаешь структуру mart-таблиц. Запусти запрос в DBeaver / DataGrip / ClickHouse client и посмотри, какие колонки есть в mart_daily_kpi.

Неделя 2: Первые задачи

Переходим от наблюдения к действию. Все задачи — под присмотром buddy с обязательным code review.

▶ Первый ad-hoc (под buddy)

Buddy выбирает простой ad-hoc запрос из бэклога (обычно помеченный good-first-issue). Примеры:

  • «Покажи top-10 игр по GGR за последний месяц в LatAm»
  • «Посчитай retention D7 для когорты FTD прошлой недели»
  • «Сравни конверсию Reg→FTD по платформам (web vs mobile)»

Результат оформляется в виде Metabase question или SQL + комментарий в Jira-тикете. Buddy ревьюит логику и SQL.

▶ Первый dbt PR

Типовая первая задача в dbt — добавить тест или исправить модель. Buddy подбирает задачу:

Пример: добавить тест на уникальность

# models/staging/betting/_betting__models.yml
models:
  - name: stg_betting__bets
    columns:
      - name: bet_id
        tests:
          - unique
          - not_null
      - name: player_id
        tests:
          - not_null
          - relationships:
              to: ref('stg_core__players')
              field: player_id

Процесс:

  • Fork / branch: feat/add-bets-tests
  • Локальный dbt test --select stg_betting__bets
  • PR → code review от buddy → merge

Подробнее о процессе: DataOps (CI/CD for data).

▶ Compliance-тренинг

В течение недели 2 проходишь обязательный compliance-тренинг:

  • Data Privacy: GDPR basics, что считается PII, как работает маскирование
  • Responsible Gambling: self-exclusion, лимиты, обязанности оператора
  • Information Security: пароли, VPN, обращение с данными

После прохождения — получаешь доступ к PII-данным (если роль требует). Доступ выдаёт Head of Compliance.

Месяц 1: Самостоятельная работа

К концу первого месяца ты должен работать самостоятельно (buddy доступен, но инициатива — твоя).

▶ Ожидания по ролям
РольК концу месяца 1
Data Engineer Самостоятельно дебажит failed DAGs, написал 1 новую stg-модель, участвует в on-call rotation (shadow)
Analyst Закрыл 2-3 ad-hoc задачи, создал 1 Metabase question, участвовал в sprint planning
Data Scientist Разобрался в Feature Store, воспроизвёл baseline модель (LTV или Churn) в Jupyter, начал EDA для новой задачи
BI Developer Создал 1 дашборд по шаблону, разобрался в semantic layer, провёл 1 сессию data literacy
▶ Участие в процессах
  • Sprint planning — активное участие: берёшь задачи, оцениваешь сложность
  • Sprint review — демонстрируешь свою работу команде
  • Retro — делишься свежим взглядом (newbie perspective ценна!)
  • Daily standup — полноценный участник: что сделал, что планирую, блокеры

Подробнее о процессах: Management Practices.

▶ Feedback-сессия с лидом

В конце месяца 1 — обязательная 1-on-1 с лидом стрима (45 мин):

  • Как прошёл onboarding: что было понятно, что нет, что можно улучшить
  • Прогресс: обзор выполненных задач, quality code review, feedback от buddy
  • Ожидания на месяц 2-3: конкретные цели и зоны роста
  • Обратная связь от новичка: что удивило, что можно улучшить в процессах
Feedback новичка — один из лучших источников улучшений. Свежий взгляд замечает то, к чему команда привыкла. Всё, что озвучено — фиксируется и разбирается.
