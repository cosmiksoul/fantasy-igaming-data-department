Event Taxonomy

Единая система именования событий, стандартная схема, tracking plan и процесс добавления новых событий.

Naming Convention

Все события следуют формату: {domain}_{object}_{action}. Это обеспечивает единообразие, сортировку и фильтрацию.

Domains

DomainDescriptionПримеры
playerДействия игрокаplayer_bet_placed, player_deposit_completed
sessionСессия и навигацияsession_page_viewed, session_started
marketingМаркетинг и кампанииmarketing_campaign_clicked, marketing_banner_viewed
systemСистемные событияsystem_error_occurred, system_maintenance_started

Objects

bet, deposit, withdrawal, bonus, session, page,
registration, kyc, game, promo, notification

Actions

placed, completed, started, viewed, settled,
issued, expired, cancelled, failed, clicked
Правило: если не можете описать событие в формате domain_object_action — скорее всего, это не одно событие, а два.

Event Schema

Стандартная структура каждого события. Все поля обязательны кроме event-specific.

▶ Required fields required
FieldTypeDescription
event_nameStringИмя события (naming convention)
event_idUUIDУникальный ID события
player_idStringID игрока (null для анонимных)
timestampISO 8601Время события с timezone
session_idStringID текущей сессии
platformStringweb / ios / android
▶ Context fields (auto-collected) auto
FieldTypeDescription
geo_ipStringСтрана по IP
device_typeStringmobile / tablet / desktop
app_versionStringВерсия приложения (для mobile)
page_urlStringURL страницы (для web)
user_agentStringUser-Agent браузера
▶ Event-specific fields varies

Зависят от типа события. Примеры:

  • player_bet_placed: bet_amount, product, game_id, odds, bet_type
  • player_deposit_completed: amount, currency, payment_method, is_first_deposit
  • session_page_viewed: page_name, referrer, load_time_ms

Процесс изменений

Как добавить новое событие или изменить существующее. Каждое изменение проходит ревью.

1. Request
Продакт/аналитик создаёт тикет в Jira с описанием: зачем нужно событие, когда должно срабатывать, какие параметры. Шаблон: event_name, trigger, properties, source.
2. Review
Ревью двумя сторонами: Analytics (бизнес-смысл, naming convention, дубликаты) + DE (техническая реализуемость, нагрузка, формат данных). Согласование за 2 рабочих дня.
3. Spec
Финальная спецификация: JSON schema, trigger condition, validation rules. Добавляется в Tracking Plan (DataHub). Версионирование: v1, v2, etc.
4. Implementation
Backend/frontend разработчик реализует отправку события. PR с тегом [tracking] для фильтрации.
5. QA
Проверка на staging: событие приходит, формат верный, все обязательные поля заполнены. Аналитик валидирует в Kafka UI / ClickHouse.
6. Deploy
Деплой в production. Мониторинг первые 24 часа: объём событий, NULL rate, ошибки парсинга. Статус в Tracking Plan: planned → live.
Deprecation: событие нельзя просто удалить. Сначала статус → deprecated, затем 30 дней grace period (для миграции потребителей), потом удаление из кода.
