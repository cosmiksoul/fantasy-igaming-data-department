Data Architecture

Полный стек данных Fantasy iGaming — от источников до обратной интеграции. Каждый слой раскрывается с описанием инструментов, альтернатив и trade-off.

Эта страница описывает целевой стейт стека Fantasy iGaming. Текущий прогресс внедрения — в разделе Roadmap.
Кликните на любой слой, чтобы увидеть инструменты. Каждый tool card содержит: что делает, почему выбран, альтернатива и ключевой trade-off.
▶ 1. Sources 6 sources

Первичные источники данных компании. Данные приходят из продакшн-систем, трекеров, CRM и внешних API. Напрямую к продакшн-базам аналитика не ходит — только через слой ingestion.

PostgreSQL production DB

Основная продакшн-база: аккаунты игроков, ставки, транзакции, балансы. Содержит текущий state всех бизнес-сущностей. Никогда не используется напрямую для аналитики — нагрузка может положить прод.

MySQL, MongoDB надёжная и зрелая, но не подходит для аналитических запросов на больших объёмах
Apache Kafka real-time

Шина реалтайм-событий iGaming-платформы. Основные топики: bets.placed, deposits.completed, sessions.events. Во время live-матчей нагрузка на bets.placed растёт в 5-10x. Три consumer groups: бэкенд (расчёт выигрышей), аналитический pipeline (стриминг в ClickHouse), fraud-сервис (real-time scoring).

RabbitMQ, Amazon Kinesis, Redpanda мощный, но требует DevOps-экспертизы для эксплуатации кластера
GA4 + GTM web analytics

Веб-аналитика и маркетинговая атрибуция. В iGaming GA4 — не основной источник аналитики, а дополнение. Продуктовая аналитика живёт в DWH на базе собственного трекинга через Kafka. GA4 используется для маркетинговой атрибуции и анализа pre-registration поведения. GTM управляет тегами на сайте.

Amplitude, Mixpanel, Snowplow бесплатный, но данные семплированы и ограничены по retention
Adjust mobile SDK

Мобильная атрибуция: установки приложений, in-app события, маркетинговая атрибуция по каналам. Критично для понимания эффективности мобильного маркетинга.

AppsFlyer, Branch хорошая интеграция с ad-сетями, но стоимость растёт с объёмом событий
CRM player lifecycle

Жизненный цикл игрока: сегментация, бонусные кампании, коммуникации. Данные о том, какие бонусы выданы, в какой сегмент попал игрок, история взаимодействий.

in-house решение, Salesforce данные часто неструктурированы и требуют очистки
External APIs feeds

Внешние фиды: коэффициенты (odds), результаты матчей, статусы платёжных провайдеров. Данные приходят через REST/WebSocket API от третьих сторон.

собственные скраперы, data vendors зависимость от сторонних провайдеров, SLA на их стороне
▶ 2. Ingestion 2 tools

Слой загрузки данных из источников в хранилище. Два подхода: batch (Airbyte) для периодической синхронизации и CDC (Debezium) для реалтайм-захвата изменений.

Airbyte primary

Основной инструмент batch-загрузки. Self-hosted, 300+ готовых коннекторов к БД, API, SaaS-сервисам. Дешевле Fivetran, но требует поддержки от DE-команды: обновления, мониторинг, custom-коннекторы.

Fivetran (managed, дороже, меньше контроля) экономия на лицензиях vs расход на ops-поддержку
Debezium / CDC real-time

Change Data Capture из продакшн PostgreSQL через Kafka Connect. Захватывает только дельты (INSERT/UPDATE/DELETE), минимальная нагрузка на прод. Критичен для near-real-time аналитики: ставки, депозиты, балансы.

AWS DMS, Striim сложнее в настройке чем batch, но данные свежее на порядок
▶ 3. Storage 2 systems

Два уровня хранения: горячее (ClickHouse для аналитики) и холодное (GCS data lake для архивации и raw-данных).

ClickHouse primary DWH

Колоночная СУБД для аналитики. Быстрые агрегации на миллиардах строк, популярна в CIS и iGaming. Self-hosted = полный контроль над данными (регуляторные требования). Основное хранилище для всех mart-таблиц.

BigQuery (serverless, дорого без партиционирования), Snowflake (западный стандарт, разделение compute/storage) быстрее и дешевле облачных DWH, но требует DE-команду для эксплуатации
GCS Data Lake cold storage

Google Cloud Storage для raw/cold данных в формате Parquet. Сырые данные хранятся здесь бессрочно. Дёшево, надёжно. Используется для re-processing, compliance (хранение 5+ лет) и ML-тренировок на исторических данных.

AWS S3, Azure Blob Storage дёшево для хранения, но запросы напрямую медленнее чем из DWH
▶ 4. Transformation dbt Core

Вся трансформация данных — через dbt Core. Три слоя моделей: staging (очистка) → intermediate (бизнес-логика) → marts (готово для BI/DS). Версионирование в Git, тесты, документация из коробки.

dbt Core primary

SQL-first фреймворк для трансформации данных в DWH. Модели организованы в три слоя, каждая модель — SQL-файл в Git. Встроенные тесты, автодокументация, lineage graph.

Spark (тяжёлые batch-задачи), SQLMesh простой для аналитиков (чистый SQL), но ограничен возможностями SQL для сложных преобразований

Staging — очистка и типизация

1:1 с источником. Приведение типов, дедупликация, переименование полей. Никакой бизнес-логики.

-- stg_betting__bets: очистка сырых ставок
SELECT
    bet_id,
    player_id,
    CAST(bet_amount AS Decimal64(2)) AS bet_amount,
    CAST(payout_amount AS Decimal64(2)) AS payout_amount,
    lower(status) AS status,
    toDateTime(placed_at) AS placed_at,
    toDateTime(settled_at) AS settled_at
FROM raw.betting_bets
WHERE bet_id IS NOT NULL

Intermediate — бизнес-логика

Джойны, расчёты, обогащение. Здесь ставка обогащается данными игрока и игры.

-- int_bets_enriched: ставка + игрок + игра
SELECT
    b.bet_id,
    b.player_id,
    p.geo,
    p.acquisition_channel,
    b.bet_amount,
    b.payout_amount,
    b.bet_amount - b.payout_amount AS ggr,
    g.game_type,
    g.provider,
    b.placed_at
FROM stg_betting__bets b
LEFT JOIN stg_players p ON b.player_id = p.player_id
LEFT JOIN stg_games g ON b.game_id = g.game_id

Marts — готовые витрины

Финальные агрегаты для BI и DS. Один mart = один бизнес-вопрос.

-- mart_daily_kpi: NGR = GGR - bonus - provider fees - payment fees
WITH daily_ggr AS (
    SELECT
        toDate(placed_at) AS dt,
        count(DISTINCT player_id) AS active_players,
        sum(bet_amount) AS turnover,
        sum(ggr) AS ggr
    FROM int_bets_enriched
    GROUP BY dt
),
daily_costs AS (
    SELECT
        dt,
        sum(bonus_cost) AS bonus_cost,
        sum(provider_fee) AS provider_fees,
        sum(payment_fee) AS payment_fees
    FROM int_daily_costs
    GROUP BY dt
)
SELECT
    g.dt,
    g.active_players,
    g.turnover,
    g.ggr,
    g.ggr - coalesce(c.bonus_cost, 0)
           - coalesce(c.provider_fees, 0)
           - coalesce(c.payment_fees, 0) AS ngr
FROM daily_ggr g
LEFT JOIN daily_costs c ON g.dt = c.dt
ORDER BY g.dt DESC
▶ 5. Orchestration Airflow

Оркестрация всех data-процессов: расписание, зависимости, ретраи, алерты. Почему не cron? Потому что cron не знает про зависимости между задачами, не умеет retry, не имеет UI для мониторинга.

Apache Airflow primary

DAG-based оркестратор. Каждый pipeline описан как Directed Acyclic Graph: extract → load → dbt run → dbt test → alert. Web UI для мониторинга, встроенные retries, SLA alerts, интеграция со Slack.

Dagster (modern, better dev UX), Prefect (cloud-native) зрелый и проверенный, но UI/DX устарели, scheduler бывает капризным

Типичный DAG

from airflow import DAG
from datetime import datetime, timedelta

default_args = {
    'owner': 'data-engineering',
    'retries': 2,
    'retry_delay': timedelta(minutes=5),
}

dag = DAG(
    'daily_pipeline',
    default_args=default_args,
    schedule_interval='0 5 * * *',  # 05:00 UTC daily
    start_date=datetime(2024, 1, 1),
)

# Task dependencies
extract_betting >> extract_payments >> extract_players \
    >> load_to_clickhouse \
    >> dbt_run_staging \
    >> dbt_run_intermediate \
    >> dbt_run_marts \
    >> dbt_test \
    >> slack_notify

Расписание: ежедневно в 05:00 UTC. Данные готовы к 07:00 для утренних отчётов CEO. При падении — 2 автоматических retry с интервалом 5 минут, затем алерт в Slack.

▶ 6. DQ & Governance 3 tools

Качество данных и управление. Три уровня защиты: dbt tests (быстрые, на каждый PR), Great Expectations (сложные валидации), DataHub (каталог и lineage). Подробнее — в разделе 06. Data Governance.

dbt tests first line

Встроенные тесты dbt: unique, not_null, accepted_values, relationships. Запускаются на каждый PR в CI и после каждого dbt run в production. Первая линия защиты от битых данных.

custom SQL tests просто и быстро, но ограничены простыми проверками
Great Expectations advanced DQ

Сложные валидации: распределения, аномалии, cross-table consistency. Профилирование данных, автоматическая генерация expectations на основе исторических данных.

Soda, Monte Carlo (managed) мощный, но кривая обучения крутая, конфигурация многословная
DataHub catalog

Каталог данных: поиск таблиц, описания полей, владельцы, lineage, теги. Self-hosted. Интеграция с dbt (автоматический импорт моделей) и Airflow.

Atlan (managed), OpenMetadata бесплатный и гибкий, но требует выделенного DE на поддержку
▶ 7. BI & Reporting 2 tools

Визуализация и self-service аналитика. Два инструмента под разные аудитории. Подробнее о дашбордах — в разделе 12. Dashboards.

Metabase primary

Основной BI-инструмент. Open-source, self-hosted. Простой интерфейс для self-service: бизнес-пользователи строят отчёты без SQL. Semantic layer через «вопросы» и «коллекции».

Tableau (мощный, дорогой), Power BI (MS-экосистема), Looker (семантический слой) бесплатный и простой, но ограничен в сложных визуализациях и embedded analytics
Looker Studio marketing

Бесплатный BI от Google. Используется маркетинговой командой: нативная интеграция с GA4, Google Ads, Sheets. Маркетинг-дашборды живут здесь.

Google Sheets (для простых отчётов) бесплатный и знакомый маркетологам, но ограниченные возможности моделирования данных
▶ 8. ML Platform 3 tools

Инфраструктура для машинного обучения: разработка моделей, эксперименты, Feature Store. Подробнее — в разделе 09. MLOps.

Python / Jupyter dev

Основная среда разработки моделей. Jupyter notebooks для EDA и прототипирования, Python-скрипты для production-кода. Стандартный стек: pandas, scikit-learn, LightGBM, PyTorch.

R (для специфических стат-задач) гибкость и экосистема, но notebooks плохо версионируются
MLflow lifecycle

Tracking экспериментов, реестр моделей, управление артефактами. Каждый эксперимент логирует параметры, метрики (AUC, MAE), версии данных. Model Registry: staging → production → archived.

Weights & Biases, Neptune.ai open-source и интегрируется с любым фреймворком, но UI базовый
Feast feature store

Feature Store: единый реестр фичей для ML. Offline store (для тренировки) + online store (для real-time inference). Пример фичи: player_avg_bet_7d — средняя ставка игрока за 7 дней.

Tecton, Hopsworks open-source, но ограниченный UI и требует ручной настройки инфраструктуры
▶ 9. Reverse ETL 1 tool

Обратная интеграция: данные из DWH отправляются в операционные системы. Замыкает цикл: аналитика → действие.

Census primary

Reverse ETL: синхронизация данных из ClickHouse в CRM, рекламные платформы, email-сервисы. Ключевой use case: ML-модель считает churn score → Census отправляет его в CRM → CRM создаёт сегмент «at risk» → игрок получает персонализированный бонус.

Hightouch, custom API-интеграции простая настройка, но стоимость растёт с количеством sync-записей
Use case: churn_score (ML) → Census → CRM segment → персонализированный бонус. Цикл: данные → модель → действие → результат → данные.
