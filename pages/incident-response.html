Incident Response

Что делать, когда данные сломались. Severity levels, протокол от обнаружения до post-mortem, и реальный пример инцидента.

Severity Levels

LevelCriteriaAcknowledgeFixPost-mortem
P1 Critical Critical mart down, CEO report broken, fraud scoring offline, data loss 30 min 4h Required (within 48h)
P2 High Non-critical dashboard stale, DQ failure on important field, partial data loss 2h 24h Required (within 1 week)
P3 Medium Minor DQ issue, docs outdated, non-critical pipeline delay, cosmetic bug 1 business day 1 sprint Optional
Severity определяется по impact, не по root cause. Сломанный staging — P3. Тот же баг в prod, если ломает CEO dashboard — P1.

Response Protocol

1. Detect
Автоматический алерт (Airflow, dbt test, Grafana) → Slack #data-alerts. Или ручное обнаружение (стейкхолдер сообщает о проблеме).
2. Acknowledge
On-call DE реагирует в thread: «investigating, ETA 30 min». Назначает severity. P1 → создаёт Slack channel #incident-YYYY-MM-DD, подключает Lead DE.
3. Diagnose
Root cause analysis: логи Airflow, ClickHouse system tables, dbt test output, source connectivity. Документирует findings в incident thread.
4. Fix
Применить fix или workaround. Приоритет: восстановить данные для потребителей. Re-run пайплайн, verify данные в mart. Hotfix через expedited PR если нужно.
5. Communicate
Уведомить стейкхолдеров: что сломалось, когда починили, есть ли data gap. Для P1: сообщение в #general и email affected stakeholders.
6. Post-mortem
Post-mortem без поиска виноватых — в Confluence. Хронология инцидента, корневая причина, что сработало хорошо, что пошло не так, action items с ответственным и дедлайном. Разбор на еженедельном командном синке.

Post-mortem — заполненный пример

mart_daily_kpi Update Failure — Champions League Final

Date2026-06-01 (Monday)
SeverityP1
Duration2h 15min (07:00 — 09:15 UTC)
ImpactCEO daily pulse dashboard показывал данные за субботу. Выходные цифры отсутствовали. CFO и CMO получили incorrect weekly summary.

Timeline

05:00 — DAG started
daily_pipeline DAG запустился по расписанию.
05:47 — Task OOM killed
Задача load_to_clickhouse убита по OOM (превышен лимит памяти 4GB). Данные за воскресенье после финала Лиги чемпионов: объём ставок в 3 раза выше обычного.
05:52 — Auto-retry #1
Airflow retry #1 — тоже OOM killed.
06:02 — Auto-retry #2
Retry #2 — OOM killed. DAG marked as failed.
07:00 — SLA alert
Airflow SLA sensor: mart_daily_kpi не обновлён к 07:00. Алерт → #data-alerts + PagerDuty.
07:12 — Acknowledged
On-call DE (Иван) acknowledged. Начал investigation.
07:30 — CEO reports issue
CEO в #general: «Цифры за выходные выглядят неправильно». Lead DE подключился к incident channel.
07:45 — Root cause found
OOM из-за всплеска данных после финала Лиги чемпионов. Память воркера 4GB, нужно 8GB+.
08:10 — Fix applied
Увеличена память воркера до 12GB. Добавлена batch-обработка (партиционирование по часам вместо полного дня).
09:00 — Pipeline re-run complete
Данные за субботу и воскресенье загружены. mart_daily_kpi обновлён.
09:15 — Resolved
Дашборд CEO проверен. Стейкхолдеры оповещены. Инцидент закрыт.

Корневая причина

Лимит памяти воркера Airflow (4GB) оказался недостаточным для всплеска данных после финала Лиги чемпионов. Объём ставок в воскресенье был в 3,2 раза выше нормы. Задача load_to_clickhouse загружает полный день одним батчем — OOM при большом объёме.

Что сработало хорошо

  • SLA alert сработал вовремя (07:00)
  • On-call DE ответил в течение 12 минут
  • Root cause найден за 33 минуты
  • Данные полностью восстановлены, без потерь

Что пошло не так

  • Нет отдельного алерта на OOM — пришёл только generic SLA miss в 07:00, а не в 05:47 при первом OOM
  • Нет autoscaling — статичный лимит памяти не справился со всплеском
  • Retry потратили время впустую — та же конфигурация, тот же результат
  • CEO обнаружил проблему раньше, чем команда успела оповестить
