MLOps

Как ML-модели проходят путь от гипотезы до production: lifecycle, эксперименты, Feature Store, мониторинг и три model cards.

ML Lifecycle

Каждая модель проходит полный цикл. Не все задачи требуют ML — если эвристика решает проблему, модель не нужна. Подробнее о стеке — в 01. Data Architecture.

1. Постановка задачи
Lead DS + Product определяют: какую бизнес-задачу решаем, какая метрика успеха, есть ли данные, оправдан ли ML (vs эвристика). Результат: ML Design Doc.
2. Подготовка данных
DS совместно с DE: выбор фичей, создание training dataset, feature engineering. Фичи регистрируются в Feature Store (Feast). Проверка качества: нет утечки данных (data leakage), корректное разделение train/test.
3. Обучение
Эксперименты в Jupyter, трекинг в MLflow: параметры, метрики, артефакты. Сравнение подходов: baseline (эвристика) → простая модель → сложная модель. Выбор по метрике + интерпретируемости.
4. Валидация
Оценка на hold-out и кросс-валидация. Проверка на смещения (bias): по гео, сегментам, VIP-тирам. Lead DS ревьюит результаты и методологию.
5. Деплой
ML Engineer упаковывает модель: Docker-контейнер + REST API (для real-time) или batch-скрипт (для offline scoring). Модель регистрируется в Model Registry: staging → production.
6. Мониторинг
Отслеживание в production: model drift (PSI), data drift (распределение входных фичей), бизнес-метрики vs baseline. Алерты при деградации.
7. Переобучение
По расписанию (monthly) или по триггеру (drift alert). Новая версия проходит валидацию → staging → production. Старая версия архивируется.

Feature Store (Feast)

Единый реестр фичей для ML. Offline store (для обучения, ClickHouse) + online store (для real-time inference, Redis). Одна фича — один источник правды.

Три типа мониторинга в production: дрифт модели, дрифт данных, бизнес-метрики.

Trade-off: precision vs recall

Для churn-модели важнее precision в верхнем дециле: мы хотим, чтобы из топ-10% «рискованных» большинство действительно ушли. Высокий recall менее критичен — лучше точно определить 40% churners, чем «поймать всех» с кучей false positives (лишние бонусы = лишние расходы).

Интеграция: churn_score → Census (Reverse ETL) → CRM segment «at risk» → персонализированный бонус или push-уведомление. Замкнутый цикл: данные → модель → действие → результат.

Архитектура

Real-time scoring через REST API (Docker + FastAPI). Фичи из online Feature Store (Redis, < 5ms). При fraud_score > 0.8 транзакция блокируется автоматически. При 0.5–0.8 — отправляется на ручную проверку в Risk-команду.

FPR критичен: каждый false positive — это легитимный игрок, которому заблокировали транзакцию. При FPR 3.2% на 100K транзакций/день = ~3,200 ложных блокировок. Цель: FPR < 5%, оптимально < 2%.

Утилизация моделей: от score к бизнес-действию

Модель в production бесполезна, если её output не приводит к конкретному действию. Для каждой из трёх моделей — как именно score превращается в бизнес-результат.

LTV Prediction — утилизация

pLTV сегментПорогДействиеИсполнитель
High LTV (>$500) top 15% Увеличить bid в paid каналах, приоритет в VIP-программе, персональный менеджер Marketing + VIP team
Medium LTV ($100–500) middle 60% Стандартная CRM-коммуникация, стандартные бонусы CRM автоматически
Low LTV (<$100) bottom 25% Снизить маркетинговый бюджет на канал, не давать крупные бонусы Marketing

Пример

Канал Facebook CIS показывает средний pLTV = $85 при CAC = $60. LTV:CAC = 1.4 — ниже порога 3.0. Решение: снизить бюджет на 50%, перераспределить в пользу Organic Search (pLTV = $320, CAC ≈ $0). Ожидаемый эффект: общий blended LTV:CAC вырастет с 2.8 до 3.5.

Churn Prediction — утилизация

Churn riskПорог scoreДействиеКаналTiming
Critical > 0.8 Персональный звонок от VIP-менеджера (для VIP) или крупный cashback Телефон / email В течение 24ч

Пример

Игрок usr_28491, сегмент regular, churn_score = 0.73. Последняя ставка 12 дней назад. Любимый спорт — футбол. CRM автоматически отправляет push: «Финал Лиги Чемпионов через 3 дня — вот фрибет $10 на любой исход». Retention rate в этом сегменте после такой кампании: +18%.

Fraud Detection — утилизация

Fraud scoreПорогДействиеАвтоматизация
Block > 0.8 Транзакция заблокирована, аккаунт flagged для расследования Полностью автоматически
Review 0.5–0.8 Транзакция на hold, уходит в очередь Risk-команды Авто-hold, ручной review
Pass < 0.5 Транзакция проходит, логируется для обучения модели Автоматически

Пример

Новый аккаунт, зарегистрирован 2 часа назад. Первый депозит $500 (в 5x выше среднего FTD). IP совпадает с 3 другими аккаунтами. fraud_score = 0.87. Автоматическая блокировка транзакции. Risk-команда проверяет — подтверждённый мульти-аккаунт. Все связанные аккаунты заблокированы.

Пороги — не константа. Их калибруют ежеквартально на основе precision/recall trade-off и бизнес-стоимости ошибок. False positive в fraud = потеря легитимного игрока ($200+ LTV). False negative = фрод-потери ($50–500 per case). Баланс пересматривается на Governance Council.
