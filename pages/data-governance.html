Data Governance

Кто владеет данными, как они каталогизированы, кто имеет доступ, как проверяется качество и сколько данные живут.

Roles & Ownership

Data Owners

Бизнес-руководители, принимающие решения о данных своего домена: доступ, качество, retention, определения метрик.

DomainOwnerOwnsDecides
Betting Head of Product fct_bets, dim_games, dim_sports_events Определения bet-метрик, доступ к raw betting data
Finance CFO mart_daily_kpi (revenue), fct_deposits, fct_withdrawals Формулы NGR/GGR, финансовая отчётность, аудит
Marketing CMO mart_marketing_performance, attribution data Модели атрибуции, доступ к маркетинговым расходам
Player PII Head of Compliance dim_players (PII fields), KYC data Доступ к PII, retention PII, GDPR requests

Data Stewards

Аналитик или DE, назначенный на домен. Отвечает за операционное качество: мониторит свежесть, чинит аномалии, поддерживает документацию.

Пример: Steward для mart_daily_kpi — Financial Analyst.

  • Проверяет данные каждое утро к 07:30 UTC
  • Валидирует NGR: сравнивает с платёжным отчётом
  • Эскалирует расхождения > 0.5% в #data-alerts
  • Обновляет документацию при изменении формул

Governance Council

Состав: Director of Data (chair), все Data Owners, Lead Analyst, Lead DE.

Cadence: Ежеквартально, 1.5 часа.

Типичная повестка:

  • Ревью DQ score по доменам
  • Новые определения метрик (спорные случаи)
  • Запросы на доступ к PII
  • Изменения retention policy

Пример решения: Q1 2026 — утверждение новой формулы Churn Rate (с 30 на 45 дней для sports-only игроков, на основе анализа сезонности).

Централизованный каталог: поиск таблиц, описания полей, владельцы, lineage. Self-hosted, интеграция с dbt и Airflow.

Data Dictionary

Стандартизированные определения ключевых терминов. Единый источник правды — если два человека спорят о значении метрики, побеждает Data Dictionary.

Процесс изменения определений

1. Proposal
Аналитик или Product создаёт RFC в Confluence: текущее определение, предлагаемое, обоснование, impact analysis.
2. Review
Data Owner домена + Lead Analyst ревьюят. Если cross-domain — выносится на Governance Council.
3. Approve & Update
Обновление в Data Dictionary, DataHub, dbt docs. Уведомление в #data-announcements. Grace period 2 недели для даунстрим-потребителей.

Access Policies (RBAC)

Role-Based Access Control. Принцип least privilege: каждая роль получает только необходимый доступ.

Role Betting Finance Player PII Marketing
Analyst read read masked read
Data Scientist read no masked read
BI Developer read read no read
Lead read read masked read
Director full full full full
Masked = PII-поля (email, phone, name) заменяются хэшем. player_id доступен для джойнов, но идентификация невозможна. Доступ к raw PII — только через запрос к Head of Compliance с обоснованием.

Row-Level Security — пример

-- Marketing Analyst видит только marketing-related данные
CREATE ROW POLICY marketing_only ON mart_marketing_performance
FOR SELECT
USING currentUser() IN ('marketing_analyst_1', 'marketing_analyst_2')
TO marketing_team;

DQ Framework

5 измерений качества данных. Каждое измерение — набор автоматизированных тестов (dbt tests + Great Expectations). Подробнее о стеке — 01. Data Architecture.

▶ Completeness dimension

Полнота данных — отсутствие NULL в обязательных полях.

# dbt test: payment_method completeness
- name: fct_deposits
  columns:
    - name: payment_method
      tests:
        - not_null:
            config:
              severity: warn
              warn_if: "> 1%"
              error_if: "> 5%"

Target: % NULL в payment_method < 1%

▶ Accuracy dimension

Точность — данные соответствуют реальности.

-- Great Expectations: actual RTP vs theoretical
SELECT
    game_id,
    abs(actual_rtp - theoretical_rtp) AS rtp_diff
FROM mart_game_performance
WHERE total_bets > 10000
  AND abs(actual_rtp - theoretical_rtp) > 0.02

Target: actual RTP vs theoretical ±2pp для игр с 10K+ ставок

▶ Timeliness dimension

Своевременность — данные обновляются по расписанию.

-- Airflow sensor: mart updated by 7:00 UTC
SELECT
    table_name,
    last_updated_at,
    dateDiff('minute', last_updated_at, now()) AS staleness_min
FROM meta.table_freshness
WHERE staleness_min > 120 -- stale > 2 hours

Target: critical marts обновлены к 07:00 UTC, staleness < 2 часов

▶ Consistency dimension

Согласованность — одни и те же данные совпадают между таблицами.

-- Cross-table consistency check
SELECT
    abs(
        (SELECT sum(bet_amount) FROM fct_bets WHERE toDate(placed_at) = today() - 1)
        - (SELECT turnover FROM mart_daily_kpi WHERE dt = today() - 1)
    ) / nullIf(
        (SELECT turnover FROM mart_daily_kpi WHERE dt = today() - 1), 0
    ) * 100 AS diff_pct

Target: SUM(fct_bets.amount) = mart_daily_kpi.turnover ±0.01%

▶ Uniqueness dimension

Уникальность — нет дубликатов по primary key.

# dbt test: no duplicates on bet_id
- name: fct_bets
  columns:
    - name: bet_id
      tests:
        - unique
        - not_null

Target: 0 дубликатов на bet_id, deposit_id, player_id

Data Lifecycle

Retention policy: данные проходят три стадии хранения. Регуляторные требования: betting data 5+ лет (Curaçao license).

Hot — 3 месяца

Storage: ClickHouse (primary SSD)

Активные данные для daily operations, дашбордов, ad-hoc запросов. Максимальная скорость. Все mart-таблицы, свежие факты.

Warm — 1 год

Storage: ClickHouse (cold storage / HDD)

Исторические данные для когортного анализа, YoY сравнений, ML-тренировок. Запросы медленнее, но доступны через тот же SQL.

Cold — 5+ лет

Storage: GCS (Parquet format)

Архив для compliance и аудита. Регуляторные требования Curaçao: betting data хранится минимум 5 лет. Доступ через ad-hoc Spark/DuckDB job.

PII retention: Player PII (email, phone, name) удаляется через 30 дней после запроса на удаление (GDPR). player_id остаётся для аналитики в анонимизированном виде.
