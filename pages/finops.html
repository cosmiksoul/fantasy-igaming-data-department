FinOps / Cost Model

Сколько стоит data-функция Fantasy iGaming и как оптимизировать затраты. Инфраструктура, SaaS, люди — полная раскладка.

Все цифры — реалистичные оценки для компании масштаба Fantasy iGaming (~500K MAU, ~$5M NGR/мес). Фактические затраты зависят от региона, провайдера и условий контракта.

Оптимизация затрат

Экономия без потери качества — постоянный процесс. Основные рычаги:

▶ ClickHouse: партиционирование и TTL

Крупнейшая статья инфраструктуры. Два главных рычага:

Партиционирование

Все fact-таблицы партиционированы по месяцам. Запросы за последний месяц не сканируют исторические данные — экономия IO и CPU.

CREATE TABLE fct_bets (
    bet_id UUID,
    player_id UInt64,
    bet_amount Decimal(18,2),
    placed_at DateTime
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(placed_at)
ORDER BY (player_id, placed_at);

TTL (Time-to-Live)

Автоматическое перемещение старых данных на дешёвое хранилище:

ALTER TABLE fct_bets
    MODIFY TTL
        placed_at + INTERVAL 3 MONTH TO VOLUME 'hot',     -- NVMe SSD
        placed_at + INTERVAL 12 MONTH TO VOLUME 'cold',   -- HDD
        placed_at + INTERVAL 5 YEAR DELETE;                -- Archive to GCS first

Экономия: ~30% стоимости хранения (HDD в 5-8x дешевле NVMe).

▶ Materialized Views

Часто используемые агрегаты вычисляются один раз при вставке, а не при каждом запросе:

CREATE MATERIALIZED VIEW mv_daily_ggr
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, geo, product)
AS SELECT
    toDate(placed_at) AS date,
    geo,
    product,
    sum(bet_amount) AS total_bets,
    sum(payout_amount) AS total_payouts,
    sum(bet_amount) - sum(payout_amount) AS ggr
FROM fct_bets
GROUP BY date, geo, product;

Экономия: дашборды грузятся в 10-50x быстрее, CPU cluster использование -40% в пиковые часы.

▶ Query Optimization

Top-3 частых проблем и их решения:

ПроблемаСимптомРешениеЭкономия
SELECT * на больших таблицах Timeout, high memory Выбирать только нужные колонки ~60% IO
JOIN без предварительной фильтрации Slow query, OOM Фильтр по дате ДО join ~80% CPU
Отсутствие LIMIT на ad-hoc Огромный result set Default LIMIT 10000, query governor Prevents OOM
Раз в квартал — «Query Review Day»: анализируем system.query_log, находим top-20 самых тяжёлых запросов, оптимизируем или кешируем.

▶ Как защитить бюджет перед CFO

CFO не разбирается в ClickHouse и dbt. Говори на языке бизнеса:

Фреймворк ROI

  • Revenue impact: «LTV-модель увеличила retention целевого сегмента на 4% → +$240K NGR/год»
  • Cost avoidance: «Self-hosted ClickHouse vs BigQuery экономит $180K/год при нашем объёме»
  • Efficiency: «Self-service BI сократил ad-hoc запросы на 60% → высвободили 1.5 FTE аналитиков на research»
  • Risk mitigation: «DQ framework предотвращает инциденты типа wrong CEO report → репутационный и операционный риск»

▶ Квартальный cost review

Раз в квартал Director of Data проводит cost review:

  • Actual vs Budget: отклонения > 10% — разбор причин
  • Cost per query: тренд средней стоимости запроса в ClickHouse
  • Unused resources: idle instances, unused SaaS seats, stale pipelines
  • Growth forecast: проекция затрат на следующий квартал с учётом роста данных

Результат — memo для CFO с рекомендациями (оптимизация, capacity planning, hiring needs).
