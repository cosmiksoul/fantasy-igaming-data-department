CI/CD for Data

Каждое изменение в dbt-моделях или пайплайнах проходит полный цикл: от feature branch до production.

1. Feature Branch
Разработчик создаёт ветку от develop. Именование: feature/DE-123-add-player-segment. Локальная разработка в dbt с dbt run --select model_name.
2. Pull Request
PR в develop. Автоматически запускается CI: dbt compile + dbt test на staging. Линтер SQL (sqlfluff). PR template: что изменено, почему, impact.
3. Code Review
Минимум 1 approve от Senior/Lead DE. Чеклист: соглашения по именованию, тесты покрывают новые поля, документация обновлена, нет ломающих изменений.
4. Merge → Staging
Merge в develop. Автоматический dbt run на staging. Проверка: данные в staging-дашборде корректны.
5. Release → Production
PR из develop в main. Release tag. dbt run на prod через Airflow. Мониторинг первые 24 часа: объём данных, DQ тесты, скорость запросов.
6. Monitoring
Grafana + Airflow SLA: свежесть данных, объём строк, время выполнения, DQ score. Алерты в Slack #data-alerts.

On-call

DE on-call ротация: 1 неделя. Все DE и Platform Engineer участвуют (4 человека в ротации, каждый дежурит раз в месяц).

Runbook — типичные инциденты

▶ Airflow DAG failed P1/P2
  1. Открыть Airflow UI → найти failed task
  2. Прочитать логи — определить root cause (OOM, timeout, source unavailable)
  3. Retry задачи через UI (Clear → Run)
  4. Если retry не помог → проверить source connectivity, ClickHouse health
  5. Если не ясно → эскалация на Lead DE в Slack #data-incidents
  6. После фикса: проверить данные в mart, уведомить потребителей
▶ ClickHouse query timeout P2
  1. Проверить system.query_log — какой запрос тормозит
  2. Проверить system.merges — не идёт ли heavy merge
  3. Kill проблемный запрос если блокирует других
  4. Если проблема в модели — оптимизировать (добавить WHERE, PREWHERE, partitioning)
  5. Если системная — эскалация на Platform Engineer
▶ Source data delayed / missing P2
  1. Проверить Airbyte sync status — последний успешный sync
  2. Проверить source connectivity (может быть downtime у провайдера)
  3. Если source down → уведомить потребителей о задержке, поставить ETA
  4. Если Airbyte issue → restart sync, проверить логи коннектора
  5. Если данные пришли частично → пометить в DQ dashboard, не давать уйти в mart
▶ DQ test failure in production P2/P3
  1. Определить severity: critical mart (P1) vs non-critical field (P3)
  2. Проверить масштаб: сколько строк затронуто, с какого момента
  3. Если данные уже в mart → откатить к последнему clean state если возможно
  4. Root cause: изменение в source? Баг в dbt model? Schema drift?
  5. Fix → re-run → verify → уведомить потребителей
