Data Lineage

Полный путь данных от источника до потребителя. Два примера: как считается NGR и как работает churn prediction pipeline.

Data lineage отвечает на вопрос: «откуда взялось это число в дашборде?». Каждый шаг — инструмент, владелец и DQ-проверки. Подробнее о стеке — в 01. Data Architecture.

Пример 1: путь NGR от ставки до дашборда CEO

Полный end-to-end: игрок делает ставку → данные проходят 8 этапов → CEO видит NGR на утреннем дашборде.

flowchart LR
    A["Betting Engine\n(ставка размещена)"] --> B["Kafka\nbets.placed"]
    B --> C["Airbyte\n(batch load)"]
    C --> D["ClickHouse\nraw.betting_bets"]
    D --> E["stg_betting__bets\n(dbt staging)"]
    E --> F["int_bets_enriched\n(dbt intermediate)"]
    F --> G["int_daily_costs\n(бонусы + fees)"]
    G --> H["mart_daily_kpi\n(NGR = GGR - costs)"]
    H --> I["Metabase\nCEO Daily Pulse"]

    style A fill:#534AB7,color:#fff,stroke:#534AB7
    style I fill:#1D9E75,color:#fff,stroke:#1D9E75
▶ Betting Engine → Kafka источник
Что происходитИгрок размещает ставку. Бэкенд записывает событие bets.placed в Kafka topic.
ИнструментApache Kafka (event streaming)
ВладелецBackend-команда
DQ-проверкиSchema validation на уровне producer (Avro schema registry)
LatencyМиллисекунды
▶ Kafka → ClickHouse raw ingestion
Что происходитAirbyte читает из Kafka и загружает в raw.betting_bets таблицу ClickHouse. Batch sync каждые 15 минут.
ИнструментAirbyte (Kafka → ClickHouse connector)
ВладелецData Engineering
DQ-проверкиМониторинг объёма: алерт если row count отклоняется > 20% от среднего
▶ raw → stg_betting__bets dbt staging
Что происходитОчистка: приведение типов, lower(status), дедупликация по bet_id, фильтрация NULL.
Инструментdbt Core (SQL model)
ВладелецAnalytics Engineer
DQ-проверкиunique(bet_id), not_null(bet_id, player_id, bet_amount)
▶ stg → int_bets_enriched dbt intermediate
Что происходитОбогащение: JOIN с dim_players (geo, segment) и dim_games (provider, game_type). Расчёт GGR = bet_amount - payout_amount.
Инструментdbt Core
ВладелецAnalytics Engineer
DQ-проверкиrelationships(player_id → dim_players), accepted_values(product, ['casino','sports'])
▶ int → mart_daily_kpi dbt mart
Что происходитАгрегация по дням: SUM(GGR), вычет бонусов и комиссий из int_daily_costs. Финальная формула: NGR = GGR - bonus_cost - provider_fees - payment_fees.
Инструментdbt Core
ВладелецFinancial Analyst (steward), CFO (data owner)
DQ-проверкиConsistency: SUM(fct_bets.amount) = mart_daily_kpi.turnover ±0.01%. Freshness: обновление к 07:00 UTC.
▶ mart → Metabase (CEO Daily Pulse) BI
Что происходитMetabase запрашивает mart_daily_kpi и отображает: NGR, GGR, Active Players, Deposits. Sparklines WoW, разбивка по гео.
ИнструментMetabase (self-hosted)
ВладелецBI Engineering
DQ-проверкиDashboard load time < 5 сек. Данные не старше 1 часа после обновления DWH.
ПотребительCEO, CFO, C-level (ежедневный утренний обзор)

Пример 2: как считается LTV игрока

Более универсальный пример — LTV как метрику считают в любом бизнесе, не только в iGaming. Полный путь: от сырых транзакций до решения «масштабировать канал или отключить».

flowchart LR
    A["fct_bets\nfct_deposits\nfct_bonuses"] --> B["int_bets_enriched\n(GGR per player)"]
    B --> C["int_daily_costs\n(бонусы + fees\nper player)"]
    C --> D["mart_player_lifetime\n(cumulative NGR,\ndays_active, segment)"]
    D --> E["Feature Store\n+ LTV Model\n(LightGBM)"]
    E --> F["mart_marketing_\nperformance\n(LTV:CAC per channel)"]
    F --> G["Metabase\nMarketing Dashboard"]

    style A fill:#534AB7,color:#fff,stroke:#534AB7
    style E fill:#D4537E,color:#fff,stroke:#D4537E
    style G fill:#1D9E75,color:#fff,stroke:#1D9E75
▶ fct_bets + fct_deposits + fct_bonuses источники
Что происходитТри факт-таблицы содержат все транзакции игрока: ставки (суммы, продукт, результат), депозиты (суммы, метод, FTD-флаг) и бонусы (тип, сумма, статус отыгрыша).
ИнструментClickHouse (DWH)
ВладелецData Engineering
DQ-проверкиunique(bet_id), unique(deposit_id), unique(bonus_id). Полнота: NULL rate < 1% в ключевых полях.
▶ int_bets_enriched → int_daily_costs dbt intermediate
Что происходитint_bets_enriched рассчитывает GGR per player (bet_amount - payout_amount). int_daily_costs агрегирует бонусные расходы и комиссии провайдеров/платёжных систем per player per day.
Инструментdbt Core
ВладелецAnalytics Engineer
DQ-проверкиConsistency: SUM(ggr) по int_bets_enriched = SUM(ggr) по mart_daily_kpi за тот же период (±0.01%)
▶ mart_player_lifetime dbt mart
Что происходит累ulative метрики по каждому игроку: lifetime_ngr (= GGR - bonus_cost - fees), days_active, total_bets, segment. Это фактический (когортный) LTV — то, сколько игрок уже принёс.
Инструментdbt Core
ВладелецFinancial Analyst (steward)
DQ-проверкиnot_null(player_id, lifetime_ngr). Бизнес-правило: lifetime_ngr не может быть отрицательным для когорт старше 6 месяцев (если да — аномалия).
▶ Feature Store + LTV Model ML
Что происходитФичи из Feast (avg_bet_7d, deposit_frequency, session_count и др.) подаются в LightGBM-модель. На выходе — прогнозный LTV на 12 месяцев. Batch scoring ежедневно к 09:00 UTC.
ИнструментFeast (Feature Store) + LightGBM + MLflow
ВладелецData Science
DQ-проверкиModel drift: PSI < 0.2. Бизнес-check: средний predicted LTV не отклоняется > 20% от когортного LTV за тот же период.
▶ mart_marketing_performance dbt mart
Что происходитPredicted LTV агрегируется по каналам и кампаниям. Рассчитывается LTV:CAC ratio = avg(predicted_ltv) / CAC. Это ключевая метрика для решений по маркетинговому бюджету.
Инструментdbt Core
ВладелецMarketing Analyst (steward), CMO (data owner)
DQ-проверкиLTV:CAC не может быть отрицательным. Алерт если LTV:CAC < 1.0 для канала с spend > $10K/мес.
▶ Metabase Marketing Dashboard BI
Что происходитCMO и маркетинг-команда видят LTV:CAC по каналам, ROAS, тренды. Решение: масштабировать канал (LTV:CAC > 3), оптимизировать (2-3) или отключить (< 1.5).
ИнструментMetabase
ВладелецBI Engineering
DQ-проверкиДанные не старше 24 часов. Dashboard load < 5 сек.
ПотребительCMO, Marketing Analysts, Growth-команда
LTV — прогноз, не факт. Когортный LTV (фактический) считается из mart_player_lifetime. Прогнозный LTV — из ML-модели. Маркетинг использует прогнозный для быстрых решений, финансы — когортный для отчётности. Оба должны сходиться на горизонте 6+ месяцев, иначе модель врёт.

Пример 3: Churn Prediction Pipeline

Более короткий пример: от факт-таблиц до персонализированного бонуса в CRM.

flowchart LR
    A["fct_bets\nfct_deposits\nfct_sessions"] --> B["Feature Store\n(Feast)"]
    B --> C["ML Model\n(LightGBM)"]
    C --> D["churn_score\n(0.0 — 1.0)"]
    D --> E["Census\n(Reverse ETL)"]
    E --> F["CRM\nсегмент 'at risk'"]
    F --> G["Персональный\nбонус / push"]

    style A fill:#534AB7,color:#fff,stroke:#534AB7
    style C fill:#D4537E,color:#fff,stroke:#D4537E
    style G fill:#1D9E75,color:#fff,stroke:#1D9E75
ЭтапИнструментВладелецОписание
Факт-таблицы ClickHouse DE fct_bets, fct_deposits, fct_sessions — сырые данные о поведении игрока
Feature Store Feast DS + DE Агрегированные фичи: avg_bet_7d, days_since_last_bet, deposit_frequency и др.
ML-модель LightGBM + MLflow Data Science Batch scoring: каждый игрок получает churn_score от 0 до 1. Ежедневно к 09:00.
Reverse ETL Census DE churn_score отправляется из ClickHouse в CRM. Sync каждые 2 часа.
CRM-действие CRM CRM / Marketing Игроки с score > 0.7 попадают в сегмент «at risk». Автоматический триггер: персональный бонус или push-уведомление.
Замкнутый цикл: данные → модель → действие → результат → данные. Если бонус сработал и игрок остался — это новые данные для следующего переобучения модели.
