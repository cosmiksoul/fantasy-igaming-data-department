Glossary / Словарь терминов

Все термины проекта в одном месте. Краткое определение + ссылка на раздел, где термин раскрыт подробнее.

Термины отсортированы по алфавиту (A-Z). Кликните на букву, чтобы раскрыть список.
▶ A
A/B Testing
Метод контролируемого эксперимента: аудитория делится на группы (control / treatment), одной показывают изменение, другой — нет. Сравнивается влияние на целевую метрику. A/B Testing
Acquisition
Привлечение новых пользователей. В iGaming — путь от первого визита до первого депозита (FTD). Измеряется через CPA, CAC, конверсионную воронку. Metrics Dictionary
Active Player
Игрок, совершивший хотя бы одну ставку за последние 30 дней. Основа для расчёта DAU/WAU/MAU. Определение зафиксировано в Data Dictionary. Data Governance
Adjust
Mobile measurement partner (MMP) — SDK для атрибуции мобильных установок, in-app событий и маркетинговых кампаний. Альтернатива: AppsFlyer. Data Architecture
Airbyte
Open-source платформа для ingestion данных. 300+ коннекторов, self-hosted. Загружает данные из источников в DWH. Альтернатива: Fivetran (managed, дороже). Data Architecture
Airflow (Apache Airflow)
Оркестратор пайплайнов данных. Управляет DAGs — направленными ациклическими графами задач. Обеспечивает расписание, retry, мониторинг, алерты. Data Architecture
Analytics Engineer
Роль на стыке DE и Analytics. Отвечает за dbt-модели, data quality, семантический слой. Работает с данными после ingestion — до BI. Org Structure
Attribution
Определение источника привлечения пользователя (какая кампания, канал, креатив привёл игрока). Ключевое для маркетинговой аналитики и расчёта ROAS. Metrics Dictionary
AUC (Area Under Curve)
Метрика качества бинарного классификатора. Площадь под ROC-кривой. Чем ближе к 1.0, тем лучше модель отличает классы. В проекте: Churn AUC 0.84, Fraud AUC 0.91. MLOps
▶ B
Batch Processing
Обработка данных порциями по расписанию (например, daily). Противоположность real-time / streaming. Большинство пайплайнов Fantasy iGaming — batch (ежедневный dbt run). DataOps
BI (Business Intelligence)
Инструменты и практики визуализации данных для принятия бизнес-решений. В проекте: Metabase (primary), Looker Studio (marketing). Data Architecture
BigQuery
Serverless columnar DWH от Google Cloud. Альтернатива ClickHouse. Плюс: не нужен ops. Минус: дороже при масштабе без правильного partitioning. Data Architecture
Bonus Cost
Затраты на бонусы игрокам (welcome bonus, cashback, freebet, freespin). Вычитается из GGR при расчёте NGR. Типовой benchmark: 10-15% от GGR. Metrics Dictionary
Bonus Cost Ratio
Отношение bonus cost к GGR (в %). Показывает, какая доля дохода уходит на бонусы. Если >20% — сигнал для пересмотра бонусной политики. Metrics Dictionary
Buddy
Опытный сотрудник, назначаемый новичку на период onboarding (обычно 1 месяц). Помогает с контекстом, отвечает на вопросы, ревьюит первые PR. Onboarding
▶ C
CAC (Customer Acquisition Cost)
Стоимость привлечения одного платящего клиента. Total Marketing Spend / Number of FTDs. Ключевая метрика unit economics. Metrics Dictionary
CDC (Change Data Capture)
Технология захвата изменений из production БД в реальном времени. В проекте: Debezium читает WAL PostgreSQL через Kafka Connect. Минимальная нагрузка на prod. Data Architecture
Census
Инструмент Reverse ETL — синхронизирует данные из DWH обратно в операционные системы (CRM, рекламные платформы). Use case: churn score → CRM → персональный бонус. Data Architecture
Churn Rate
Доля игроков, прекративших активность за период. Обычно: нет ставок 30+ дней = churned. Complement к retention. Metrics Dictionary
ClickHouse
Колоночная СУБД для аналитических запросов. Быстрые агрегации на больших объёмах. Primary DWH в проекте. Self-hosted, популярен в CIS/iGaming. Data Architecture
Cohort Analysis
Группировка пользователей по дате первого действия (регистрация, FTD) и отслеживание поведения когорты во времени. Основа для retention heatmap и LTV-прогнозов. Dashboards
Confidence Interval (CI)
Диапазон значений, в который с заданной вероятностью (обычно 95%) попадает истинное значение параметра. Используется в отчётах A/B-тестов для оценки effect size. A/B Testing
Conversion
Переход пользователя на следующий этап воронки. В iGaming ключевые: Visit→Registration, Registration→FTD, FTD→Repeat Deposit. Metrics Dictionary
CPA (Cost Per Acquisition)
Стоимость привлечения одного целевого действия. В iGaming обычно = CPD (Cost Per Deposit, стоимость за FTD). Metrics Dictionary
CRM (Customer Relationship Management)
Система управления жизненным циклом игрока: сегментация, коммуникации, бонусы, retention campaigns. Источник данных для analytics и потребитель ML-скоров через Reverse ETL. Data Architecture
Cross-validation
Техника оценки ML-модели: данные разбиваются на K фолдов, модель обучается на K-1 и проверяется на оставшемся. Снижает риск overfitting. MLOps
▶ D
DAG (Directed Acyclic Graph)
Направленный ациклический граф — структура пайплайна в Airflow. Определяет последовательность задач и зависимости между ними. Data Architecture
Data Catalog
Централизованный реестр всех datasets: описания таблиц, ownership, lineage, tags, popularity. В проекте: DataHub. Data Governance
Data Dictionary
Стандартизированные определения бизнес-терминов (Active Player, FTD, GGR, NGR и т.д.). Утверждается на Governance Council. Изменения через формальный процесс. Data Governance
Data Drift
Изменение распределения входных данных ML-модели по сравнению с обучающей выборкой. Может привести к деградации модели. Мониторится через KS-test. MLOps
Data Governance
Совокупность практик управления данными: ownership, quality, access control, lifecycle, catalog. Обеспечивает доверие к данным. Data Governance
Data Lake
Хранилище сырых данных в исходном формате. В проекте: GCS (Google Cloud Storage) для raw / cold данных в формате Parquet. Data Architecture
Data Lineage
Визуализация пути данных от источника до потребителя. Показывает, откуда данные пришли, как трансформировались и куда попали. Data Lineage
Data Mesh
Архитектурный подход: децентрализация ownership данных по доменам. Каждый домен владеет и обслуживает свои data products. Evaluation запланирован на Q4. Roadmap
Data Owner
Бизнес-руководитель, ответственный за данные домена. Принимает решения о доступе, качестве, retention. Пример: Head of Product — owner betting data. Data Governance
Data Quality (DQ)
Измерение качества данных по 5 измерениям: completeness, accuracy, timeliness, consistency, uniqueness. Автоматизируется через dbt tests и Great Expectations. Data Governance
Data Steward
Технический ответственный за качество конкретного dataset (обычно аналитик или DE). Проверяет freshness, DQ scores, документацию. Data Governance
DataHub
Open-source data catalog (от LinkedIn). Хранит метаданные: описания таблиц, lineage, ownership, tags, DQ scores. Альтернативы: Atlan, Alation. Data Architecture
dbt (data build tool)
Инструмент трансформации данных (T в ELT). SQL-модели в три слоя: staging → intermediate → marts. Включает тестирование, документацию, lineage. Data Architecture
Debezium
Open-source CDC-платформа. Читает transaction log (WAL) из PostgreSQL и стримит изменения в Kafka. Минимальная нагрузка на production DB. Data Architecture
DWH (Data Warehouse)
Централизованное хранилище структурированных данных, оптимизированное для аналитических запросов. В проекте: ClickHouse. Data Architecture
▶ E
ELT / ETL
Extract-Load-Transform (ELT) vs Extract-Transform-Load (ETL). В modern data stack используется ELT: данные сначала загружаются в DWH «как есть», потом трансформируются (dbt). Data Architecture
ERD (Entity-Relationship Diagram)
Диаграмма связей между сущностями (таблицами). В проекте: mermaid.js erDiagram с 9 core entities. Data Model
Event Taxonomy
Систематика именования событий: {domain}_{object}_{action}. Стандарт для tracking plan. Пример: player_bet_placed, session_page_viewed. Event Taxonomy
▶ F
Feast
Open-source Feature Store. Хранит, версионирует и сервирует фичи для ML-моделей. Пример: player_avg_bet_7d. Альтернатива: Tecton. MLOps
Feature Store
Централизованное хранилище ML-фичей. Обеспечивает consistency между training и serving, переиспользование фичей между моделями. MLOps
FinOps
Практика управления затратами на cloud-инфраструктуру и data-платформу. Оптимизация: партиционирование, TTL, reserved instances, query review. FinOps
FPR (False Positive Rate)
Доля ложных срабатываний. В fraud detection: легитимная транзакция ошибочно помечена как фрод. FPR < 3% — целевой порог. MLOps
FTD (First-Time Depositor)
Игрок, впервые внёсший депозит. Ключевой этап воронки. FTD Count — одна из главных метрик acquisition. Metrics Dictionary
▶ G
GA4 (Google Analytics 4)
Платформа веб-аналитики от Google. Используется для маркетинговой атрибуции и web-аналитики. В проекте — в связке с GTM. Data Architecture
GCS (Google Cloud Storage)
Объектное хранилище Google Cloud. Используется как Data Lake для сырых данных и cold-архивов в формате Parquet. Data Architecture
GGR (Gross Gaming Revenue)
Валовой игровой доход. Сумма всех ставок минус сумма всех выплат: Bets - Payouts. Основная top-line метрика. Metrics Dictionary
Grafana
Платформа мониторинга и визуализации. В проекте: мониторинг инфраструктуры (ClickHouse, Airflow), DQ dashboards, алерты. DataOps
Great Expectations
Open-source фреймворк для валидации данных. Комплексные проверки DQ: cross-table consistency, distribution checks, custom validations. Дополняет dbt tests. Data Architecture
GTM (Google Tag Manager)
Система управления тегами на сайте. Позволяет маркетингу добавлять/изменять tracking без участия разработчиков. Server-side GTM для конфиденциальности. Data Architecture
Guardrail Metric
Защитная метрика в A/B-тесте. Не должна ухудшиться при внедрении изменения. Пример: тестируем новую форму депозита — guardrail: 24h deposit failure rate. A/B Testing
▶ H
HADI Cycle
Фреймворк для экспериментов: Hypothesis → Action → Data → Insight, затем цикл повторяется. Основа процесса A/B-тестирования. A/B Testing
Hold %
Процент от ставок, который оператор оставляет себе. GGR / Turnover * 100%. В казино (slots) обычно 3-5%, в sports — 5-8%. Metrics Dictionary
Hypothesis
Формализованное предположение для A/B-теста. Формат: «Если [изменение], то [метрика] [направление] на [MDE], потому что [рациональ]». A/B Testing
▶ I
ICE / RICE
Фреймворки приоритизации. ICE: Impact × Confidence × Ease. RICE: Reach × Impact × Confidence / Effort. Используются для ранжирования гипотез и задач в бэклоге. A/B Testing, Management
Incident
Событие, нарушающее SLA или нормальное функционирование данных. Классифицируется по severity: P1 (critical) → P2 (high) → P3 (medium). Incident Response
▶ K
Kafka (Apache Kafka)
Распределённая платформа потоковой обработки данных. В проекте: real-time события (ставки, депозиты, сессии). Миллионы сообщений в секунду, multiple consumers. Data Architecture
KPI (Key Performance Indicator)
Ключевой показатель эффективности. Top-level KPIs Fantasy iGaming: NGR, Active Players, LTV:CAC, Retention. Metrics Dictionary
KYC (Know Your Customer)
Процедура верификации личности игрока. Обязательна по лицензии. Статусы: pending → verified → rejected. Влияет на доступ к выводу средств. Data Model
▶ L
Lakehouse
Архитектура, объединяющая Data Lake (raw / cheap storage) и Data Warehouse (structured / fast queries). В проекте: ClickHouse + GCS Parquet — гибридный подход. Data Architecture
LightGBM
Gradient boosting фреймворк от Microsoft. Быстрый, эффективный на табличных данных. В проекте: алгоритм для LTV, Churn и Fraud моделей. MLOps
LTV (Lifetime Value)
Прогнозируемый суммарный NGR от игрока за всё время жизни (или 12 месяцев). Ключевая метрика unit economics. LTV > CAC — бизнес жизнеспособен. Metrics Dictionary
LTV:CAC Ratio
Отношение LTV к CAC. Показывает окупаемость привлечения. Benchmark: >3x — отлично, <1.5x — убыточно. Metrics Dictionary
▶ M
MAE (Mean Absolute Error)
Средняя абсолютная ошибка прогноза. В проекте: MAE LTV-модели = $42.3 — в среднем прогноз отличается от реальности на $42. MLOps
MAPE (Mean Absolute Percentage Error)
Средняя абсолютная процентная ошибка. MAE в процентах от фактического значения. Целевой MAPE LTV-модели: <20%. MLOps
Mart (Data Mart)
Финальный слой dbt-моделей. BI/DS-ready таблицы: mart_daily_kpi, mart_player_lifetime, mart_cohort_analysis. Потребители: дашборды, отчёты, модели. Data Model
Materialized View
Предвычисленная агрегация, обновляемая при вставке данных. Ускоряет запросы в 10-50x. В ClickHouse — SummingMergeTree / AggregatingMergeTree. FinOps
MAU / DAU / WAU
Monthly / Daily / Weekly Active Users. Количество уникальных активных игроков за период. MAU Fantasy iGaming: ~500K. Metrics Dictionary
MDE (Minimum Detectable Effect)
Минимальный эффект, который A/B-тест способен обнаружить при заданном sample size и power. Чем меньше MDE, тем больше нужен sample. A/B Testing
Metabase
Open-source BI-платформа. Primary инструмент визуализации в проекте. Self-service: бизнес-пользователи сами строят запросы и дашборды. Data Architecture
MLflow
Open-source платформа для ML lifecycle: tracking экспериментов, model registry, артефакты. Альтернатива: Weights & Biases. MLOps
MLOps
Практики операционализации ML: CI/CD для моделей, мониторинг, retraining, feature store, model registry. DevOps для Machine Learning. MLOps
Model Drift
Деградация производительности ML-модели со временем из-за изменений в данных или бизнесе. Мониторится через PSI (Population Stability Index). MLOps
Model Registry
Централизованное хранилище версий ML-моделей: staging → production → archived. Процесс промоушена через review Lead DS и ML Engineer. MLOps
▶ N
NGR (Net Gaming Revenue)
Чистый игровой доход — то, что бизнес реально оставляет себе. GGR - Bonus Cost - Provider Fees - Payment Fees - Taxes. Главная финансовая метрика. Metrics Dictionary
▶ O
OKR (Objectives and Key Results)
Фреймворк целеполагания: Objective (качественная цель) + 3-5 Key Results (измеримые результаты). Квартальный цикл. Management
On-call
Дежурство DE-инженера (ротация по неделям). On-call реагирует на алерты, дебажит failed DAGs, эскалирует P1 инциденты. DataOps
Overfitting
Переобучение ML-модели: высокая точность на train-данных, низкая на новых. Борьба: cross-validation, regularization, early stopping. MLOps
▶ P
P-value
Вероятность получить наблюдаемый (или более экстремальный) результат при условии, что нулевая гипотеза верна. p < 0.05 — стандартный порог статистической значимости. A/B Testing
Parquet
Колоночный формат хранения данных. Эффективная компрессия, быстрое чтение отдельных колонок. Используется в Data Lake (GCS) для архивных данных. Data Architecture
Partitioning
Разбиение таблицы на физические части (обычно по дате). Запросы с фильтром по дате читают только нужные партиции. Ключевая оптимизация в ClickHouse. FinOps
Payback Period
Время окупаемости затрат на привлечение игрока. CAC / (Monthly NGR per player). Benchmark: <6 месяцев — здоровый бизнес. Metrics Dictionary
PII (Personally Identifiable Information)
Персональные данные: email, телефон, документы, IP. Доступ ограничен (RBAC), маскируется для non-compliance ролей. GDPR: удаление по запросу за 30 дней. Data Governance
Post-mortem
Документ-разбор инцидента без поиска виноватых. Содержит: timeline, root cause, что сработало, что нет, action items с дедлайнами. Incident Response
Power (Statistical Power)
Вероятность обнаружить реальный эффект в A/B-тесте (если он существует). Стандарт: 80%. Power = 1 - P(Type II Error). A/B Testing
Precision
Доля верно определённых положительных случаев среди всех, которые модель назвала положительными. TP / (TP + FP). Важна в fraud detection (минимизация false positives). MLOps
PSI (Population Stability Index)
Метрика стабильности распределения данных. Используется для мониторинга model drift. PSI > 0.2 — значительный сдвиг, нужен retrain. MLOps
▶ R
RACI
Матрица ответственности: Responsible (делает), Accountable (отвечает), Consulted (консультирует), Informed (информируется). Определена для 7 ключевых процессов. Org Structure
RBAC (Role-Based Access Control)
Управление доступом на основе ролей. Матрица: роль × домен данных → уровень доступа (full / read / masked / no). Data Governance
Recall
Доля верно найденных положительных случаев среди всех реально положительных. TP / (TP + FN). Важна в churn prediction (не пропустить churning player). MLOps
Retention
Доля пользователей, вернувшихся через N дней после первого действия. D1/D7/D30 — стандартные окна. Визуализируется cohort heatmap. Metrics Dictionary
Reverse ETL
Обратная синхронизация: данные из DWH → операционные системы (CRM, Ad platforms). Пример: churn score из ClickHouse → CRM → персонализированный бонус. Data Architecture
ROAS (Return on Ad Spend)
Возврат на рекламные инвестиции. Revenue / Ad Spend. ROAS >3x — хороший показатель для iGaming. Metrics Dictionary
Row-level Security (RLS)
Ограничение видимости строк таблицы в зависимости от роли пользователя. Пример: аналитик CIS видит только данные geo = 'CIS'. Data Governance
RTP (Return to Player)
Процент от ставок, возвращаемый игрокам в виде выигрышей (теоретический). Для слотов: 94-97%. Actual RTP мониторится vs theoretical (отклонение ±2pp — alert). Metrics Dictionary
Runbook
Пошаговая инструкция для on-call инженера: что проверить, как починить типовой инцидент. 4 runbook в проекте: DAG failure, query timeout, source delay, DQ failure. DataOps
▶ S
Sample Size
Минимальное количество наблюдений для статистически значимого результата A/B-теста. Зависит от baseline, MDE, power и significance level. A/B Testing
Semantic Layer
Слой абстракции между DWH и конечным пользователем. Стандартные определения метрик и dimensions, доступные через BI без SQL. Деплоится в Q2. Roadmap
SLA (Service Level Agreement)
Соглашение об уровне обслуживания. Матрица: кто → кому → что → целевой показатель → эскалация. Пример: DE → Analytics: данные к 07:00 UTC. SLA
Snowflake
Cloud DWH с разделением compute/storage. Западный стандарт. Альтернатива ClickHouse. Плюс: serverless, auto-scaling. Минус: дороже при больших объёмах. Data Architecture
Staging (stg_)
Первый слой dbt-моделей. 1:1 с source: type casting, dedup, cleanup. Без бизнес-логики. Пример: stg_betting__bets. Data Model
Statistical Significance
Порог, при котором результат A/B-теста считается не случайным. Стандарт: p < 0.05 (95% confidence level). A/B Testing
▶ T
Taxonomy
Систематика классификации и именования (событий, метрик, данных). В проекте: Event Taxonomy — стандарт именования tracking events. Event Taxonomy
Timeliness
Одно из 5 измерений Data Quality: данные доступны вовремя. Измеряется как «mart обновлён к 07:00 UTC». Data Governance
Tracking Plan
Таблица всех отслеживаемых событий: название, триггер, параметры, источник (frontend/backend), owner, статус. 15 событий в проекте. Event Taxonomy
TTL (Time-to-Live)
Политика автоматического перемещения или удаления данных по возрасту. В ClickHouse: hot (3 мес, NVMe) → cold (12 мес, HDD) → archive (5 лет, GCS). FinOps
Turnover (Handle)
Общая сумма всех ставок за период. Самая верхняя метрика revenue: из неё вычитаются выплаты (= GGR), затем затраты (= NGR). Metrics Dictionary
Type I / Type II Error
Type I (false positive): видим эффект, которого нет (p-value ошибка). Type II (false negative): не видим реальный эффект (недостаточный power). A/B Testing
▶ U
Unit Economics
Экономика единицы продукта (одного игрока): LTV, CAC, LTV:CAC, Payback Period. Показывает, окупается ли привлечение на уровне когорты. Metrics Dictionary
▶ W
Wagering (Wagering Requirement)
Условие отыгрыша бонуса: сколько раз нужно поставить сумму бонуса, прежде чем можно вывести выигрыш. Пример: бонус $100 с вейджером x30 = нужно поставить $3000. Data Model
Whale
VIP-игрок с аномально высоким объёмом ставок и депозитов. Обычно top 1-2% игроков генерируют 20-30% GGR. Требуют персонального подхода и мониторинга. Data Governance
